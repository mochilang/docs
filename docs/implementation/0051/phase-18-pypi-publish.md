## Gate

`TestPhase18PypiPublishPython`: 1 dedicated fixture (`publish_dryrun_trusted_publishing`) plus the GitHub Actions workflow `transpiler3-python-publish.yml`. The local Go test verifies workflow YAML shape (`id-token: write` permission, `astral-sh/setup-uv@v3` pinned, `uv publish --dry-run --trusted-publishing always` invocation). The CI gate runs the dry-run against the TestPyPI staging mirror (`https://test.pypi.org/`); a real upload happens only on a tagged release. `uv publish --dry-run` exits 0 when the OIDC exchange succeeds, the wheel + sdist signature is produced, the PEP 740 attestation is generated by sigstore 3.0+, and the upload-endpoint handshake returns HTTP 200 without committing the upload.

## Goal-alignment audit

Phase 18 is the last link in the chain. Phases 1-14 emit working Python; Phase 15 packages it as a wheel; Phase 16 makes the wheel reproducible; Phase 17 ships a Jupyter kernel; Phase 18 publishes the wheel to PyPI so anyone with `pip install mochi-<pkg>` reaches it. The chain is incomplete without publish: a Mochi-built wheel that lives only on the developer's laptop is not a distribution. Phase 18 wires uv's OIDC publish flow, sigstore signing, and PEP 740 attestations, eliminating long-lived API tokens and giving every published wheel a cryptographic provenance trail that resists 2022-2024-era supply-chain attacks.

## Sub-phase 18.0 -- GitHub Actions workflow

### Goal-alignment audit (18.0)

Trusted Publishing requires the publishing workflow to run in a GitHub Actions context that mints an OIDC token (`id-token: write` permission) and matches a configured trust (repo + workflow filename + environment) on the PyPI side. Phase 18.0 emits `.github/workflows/publish.yml` with the exact permission set, the pinned uv version, and the `environment: pypi` block so Mochi-emitted projects ship with a working publish workflow out of the box.

### Decisions made (18.0)

**Emitted workflow** at `.github/workflows/publish.yml` (template instantiated per project):

```yaml
name: Publish to PyPI

on:
  release:
    types: [published]
  workflow_dispatch: {}
  pull_request:
    paths:
      - .github/workflows/publish.yml
      - pyproject.toml

jobs:
  build:
    name: Build wheel + sdist
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
        with:
          version: "0.7.0"
          enable-cache: true
      - run: uv python install 3.12.7
      - name: Build reproducibly
        run: |
          export SOURCE_DATE_EPOCH=$(git log -1 --pretty=%ct)
          uv build
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/

  publish:
    name: Publish via Trusted Publishing
    needs: build
    runs-on: ubuntu-24.04
    if: github.event_name == 'release'
    environment: pypi
    permissions:
      id-token: write
      contents: read
      attestations: write
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/
      - uses: astral-sh/setup-uv@v3
        with:
          version: "0.7.0"
      - run: uv publish --trusted-publishing always

  publish-dryrun:
    name: TestPyPI dry-run
    needs: build
    runs-on: ubuntu-24.04
    if: github.event_name == 'pull_request'
    environment: testpypi
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/
      - uses: astral-sh/setup-uv@v3
        with:
          version: "0.7.0"
      - run: |
          uv publish --dry-run --trusted-publishing always \
            --publish-url https://test.pypi.org/legacy/
```

**`id-token: write`** is the GitHub Actions permission that gives the workflow an OIDC token via `ACTIONS_ID_TOKEN_REQUEST_URL` / `ACTIONS_ID_TOKEN_REQUEST_TOKEN`. Without it, `uv publish --trusted-publishing always` fails with `no OIDC token available`.

**`contents: read`** is the minimum read of the repo; default `contents: write` would let a compromised action push to main, which we deny.

**`attestations: write`** is needed for the sigstore attestation upload step (sub-phase 18.2).

**`environment: pypi`** pairs with the PyPI-side Trusted Publisher configuration to constrain the trust to releases that have passed any approval step configured on the environment (a "production" environment can require manual approval; the Mochi default does not but the emitted workflow supports it).

**`astral-sh/setup-uv@v3`** pinned to `version: "0.7.0"` keeps the publish step deterministic across uv minor releases.

**Workflow filename**: `publish.yml`. The filename is part of the PyPI Trusted Publisher trust; changing it breaks the OIDC match. The emitter writes this name exactly and documents the constraint.

## Sub-phase 18.1 -- uv publish OIDC flow + TestPyPI dry-run

### Goal-alignment audit (18.1)

The workflow is the carrier; `uv publish` is the actual publisher. Phase 18.1 verifies the OIDC token exchange end-to-end against TestPyPI (PyPI's mirror at `test.pypi.org`), so the gate catches credential misconfiguration before the real-PyPI publish happens on tag.

### Decisions made (18.1)

**`uv publish --trusted-publishing always`**: uv 0.7+ detects the GitHub Actions environment (via `GITHUB_ACTIONS=true` and the OIDC env vars), exchanges the OIDC token for a 15-minute PyPI publish credential, and uploads. The `always` value forces the OIDC path and fails if not in a CI context; the alternative `auto` falls back to `$UV_PUBLISH_TOKEN` which we never want for trusted publishing.

**TestPyPI mirror**: `https://test.pypi.org/legacy/` is the staging endpoint. PR CI runs `uv publish --dry-run --publish-url https://test.pypi.org/legacy/` to exercise the full token exchange without committing the upload. TestPyPI has its own Trusted Publisher configuration (set up once per project; the Mochi onboarding docs document the steps).

**`--dry-run`** is uv 0.7+ flag that performs every step except the final POST. The OIDC exchange happens; the wheel + sdist are read; sigstore signing is invoked; PEP 740 attestation is produced; the upload-endpoint HTTP handshake completes; the final `multipart/form-data` body is not transmitted. Exit code 0 means the chain is healthy.

**OIDC claim assertions** verified by PyPI on `--dry-run`:

```
sub: repo:mochilang/mochi-example-app:environment:pypi
aud: pypi
iss: https://token.actions.githubusercontent.com
repository: mochilang/mochi-example-app
workflow: publish.yml
ref: refs/tags/v0.1.0
environment: pypi
```

PyPI validates each of these matches the configured Trusted Publisher; mismatches return HTTP 403 with a structured error. The gate prints the PyPI error verbatim on failure (no token leakage).

**No long-lived secret**: there is no `PYPI_API_TOKEN` GitHub secret in the Mochi-emitted workflow. The 2022-2024 supply-chain incidents (PyPI typosquats with stolen-token uploads) motivate this hardening; the emitted workflow refuses to fall back to token auth.

## Sub-phase 18.2 -- sigstore + PEP 740 attestations

### Goal-alignment audit (18.2)

OIDC publishing prevents stolen-token abuse but does not prove that the wheel currently on PyPI matches the wheel the CI host built. PEP 740 (accepted 2024) adds sigstore-signed attestations to PyPI uploads: each wheel ships with a JSON attestation listing the GitHub workflow run id, the commit SHA, and the build command, signed by sigstore's transparency-log-backed key. Phase 18.2 wires the attestation generation into the publish step so every Mochi-emitted wheel has cryptographic provenance.

### Decisions made (18.2)

**`sigstore 3.0+`**: the Python sigstore client (https://github.com/sigstore/sigstore-python). uv 0.7+ embeds sigstore signing into `uv publish --trusted-publishing always`; the user does not invoke sigstore directly. The 3.0 floor is the version where the attestation format matches PEP 740 v1.0 (earlier 2.x releases had divergent JSON shapes).

**PEP 740 attestation envelope** (one per uploaded artifact):

```json
{
  "version": 1,
  "verification_material": {
    "certificate": "MIIC...base64...",
    "transparency_entries": [{
      "log_index": 123456789,
      "log_id": { "key_id": "..." },
      "kind_version": { "kind": "intoto", "version": "0.0.2" },
      "integrated_time": "1717000000",
      "canonicalized_body": "eyJh..."
    }]
  },
  "envelope": {
    "statement": "eyJzdWJqZWN0IjpbeyJuYW1lIjoibW9jaGlfZXhhbXBsZV9hcHAtMC4xLjAtcHkzLW5vbmUtYW55LndobCIsImRpZ2VzdCI6eyJzaGEyNTYiOiI..."
  }
}
```

The `statement` is a base64-encoded in-toto v0.0.2 statement listing the artifact name, SHA256, and predicate (PEP 740 v1.0 `https://docs.pypi.org/attestations/publish/v1` predicate type).

**Upload path**: uv 0.7+ POSTs the attestation alongside the wheel to PyPI's `/legacy/?:action=file_upload&attestation=1` endpoint. PyPI stores the attestation in the project's metadata; users verify via `uv tool run sigstore verify pypi mochi-example-app 0.1.0` (or `pip install sigstore; sigstore verify pypi ...`).

**Local artifact**: `dist/<pkg>-<ver>-py3-none-any.whl.publish.attestation` is written alongside the wheel during `uv publish`; the dry-run path writes it too. The test runner verifies the file is present, parses the JSON, and asserts the embedded SHA256 matches `shasum -a 256 dist/<pkg>-<ver>-py3-none-any.whl`.

**Transparency log**: sigstore writes the signing event to the Rekor public transparency log (https://rekor.sigstore.dev/). The log entry is queryable forever; even if the PyPI project is deleted, the provenance trail survives.

## Sub-phase 18.3 -- publish-to-PyPI dry-run gate

### Goal-alignment audit (18.3)

Sub-phases 18.0-18.2 build the publish infrastructure; 18.3 is the gate that proves it works without committing a real release. The `--dry-run` flag turns publish into a one-step health check that PR CI runs on every workflow change, surfacing OIDC misconfiguration, sigstore signing failures, or PyPI endpoint outages before a tagged release attempts a real upload.

### Decisions made (18.3)

**Gate command** (run from PR CI):

```bash
uv publish --dry-run \
    --trusted-publishing always \
    --publish-url https://test.pypi.org/legacy/ \
    dist/*.whl dist/*.tar.gz
```

**Exit code semantics**:

- `0` -- OIDC exchange succeeded, signing produced an attestation, the upload-endpoint HTTP 200 handshake completed (PyPI accepted the request shape but did not commit because of `--dry-run`).
- `2` -- OIDC unavailable (missing `id-token: write`, not in GitHub Actions).
- `3` -- sigstore signing failed (Fulcio outage, cert chain rejection, transparency log unreachable).
- `4` -- PyPI rejected the OIDC token (trust misconfiguration; the wrong `environment` or `workflow` claim).
- `5` -- PyPI rejected the artifact (name collision on TestPyPI for the same version; resolved by bumping `[project].version` in the test fixture).

The Go test wrapper maps each exit code to a tier-specific test failure with a descriptive error.

**Negative tests**:

- `TestPhase18OidcMissing`: removes the `id-token: write` permission from a synthetic workflow and asserts `uv publish` exits with code 2.
- `TestPhase18TrustMismatch`: configures the TestPyPI trust for a different workflow filename (`bad.yml`) and asserts code 4 with PyPI's "publisher mismatch" message.

**No real PyPI upload from PR CI**: the dry-run gate hits TestPyPI only. Real PyPI uploads happen exclusively from the `publish` job on a `release: published` event; that path is exercised at tag-cut time and is not part of the per-PR gate.

**`workflow_dispatch` trigger**: the emitted workflow also accepts manual dispatch with a `dry_run: true` input; this lets a maintainer rehearse the publish flow on demand without cutting a release. Useful before the first real publish.

**TestPyPI project name**: the fixture publishes to `mochi-publish-dryrun` on TestPyPI (a reserved name claimed during Phase 1 of the runtime-stub registration plan in research note 10 §"Name reservation strategy"). The version is bumped per CI run to avoid the "file already exists" rejection.

## Test set

- `TestPhase18PypiPublishPython` -- runs `uv publish --dry-run --trusted-publishing always` against TestPyPI in CI; asserts exit 0 plus presence of `dist/*.attestation`.
- `TestPhase18WorkflowShape` -- parses the emitted `publish.yml` and asserts `permissions.id-token: write`, `environment: pypi`, `astral-sh/setup-uv@v3` with pinned `version: "0.7.0"`, and `uv publish --trusted-publishing always` as the final step.
- `TestPhase18AttestationShape` -- reads the local `.attestation` file and asserts PEP 740 v1.0 envelope structure (`version: 1`, `verification_material.certificate`, `envelope.statement`) and embedded SHA256 matches the wheel SHA.
- `TestPhase18OidcMissing` (negative) -- workflow without `id-token: write`; asserts `uv publish` exits with the missing-OIDC error.
- `TestPhase18TrustMismatch` (negative) -- workflow filename does not match the configured Trusted Publisher; asserts PyPI returns HTTP 403 and uv exits with the mismatch error.

## Deferred work

- Real-PyPI publish at the first Mochi tagged release. The dry-run gate covers the CI path; the actual `pip install mochi-<pkg>` round-trip is verified at v0.1 cut, not in the PR gate.
- GitLab CI Trusted Publishing variant (the OIDC dance differs; PyPI supports both providers but the workflow YAML diverges). Out of v1.
- ActiveState and Google Cloud Trusted Publisher variants. Same rationale; v2.
- Sigstore key-less verification CLI shipped inside `mochi` itself (so users do not need to `pip install sigstore` to verify a Mochi-published wheel). Tracked as a v1.5 enhancement.
- Wheel + sdist signing via PGP (legacy path). PyPI deprecated PGP signatures in 2023; Mochi never emits them.
- Conda-forge feedstock submission alongside PyPI publish. v2; tracked as open question Q8 in research note 12.
